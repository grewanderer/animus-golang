# ADR-0008: Модель аутентификации и сессий

**Статус:** Принято

**Дата:** 2026-02-03

## Контекст
Платформа требует SSO‑аутентификации (OIDC как основной вариант, SAML как опция), строгого контроля времени жизни сессии, принудительного logout и ограничения числа активных сессий. При этом нельзя хранить токены в явном виде, а все события аутентификации должны быть аудируемыми.

## Решение
- В шлюзе внедряется единая auth‑абстракция с режимами OIDC, SAML, dev и disabled.
- Сессии сохраняются в `auth_sessions` с TTL, отметками last‑seen и ревокациями; хэш ID‑токена сохраняется вместо токена.
- Вводится `POST /auth/force-logout` для административной ревокации сессий или всех сессий субъекта.
- Ограничение числа активных сессий пользователя реализуется в `SessionManager`.
- Эмиссия audit‑событий: `auth.user_logged_in`, `auth.user_logged_out`, `auth.session_expired`, `auth.forced_logout`.

## Обоснование
- Централизация auth в шлюзе гарантирует единые правила и не требует хранения секретов в сервисах CP.
- Персистентные сессии с TTL обеспечивают воспроизводимую проверку доступа и аудит.
- Хэш токена минимизирует риск утечки при сохранении возможности корреляции.

## Последствия
- Требуется БД для хранения сессий и ревокаций.
- Токены не доступны для последующей выдачи или экспорта.
- Любая ошибка управления сессиями фиксируется в аудите.
