# ADR-0009: Точки enforcement RBAC

**Статус:** Принято

**Дата:** 2026-02-03

## Контекст
RBAC по матрице ролей проектного уровня должен применяться ко всем операциям CP, включая чтение, запись, download/export. Нужны явные и аудируемые точки проверки доступа без неявного состояния.

## Решение
- Шлюз выполняет аутентификацию и проксирует идентичность в CP через подписанные заголовки.
- Каждый сервис CP использует middleware `auth.Middleware` + `rbac.Authorizer`.
- Project‑scoping обеспечивается `ProjectResolver`, который извлекает `project_id` из пути/заголовков/контекста.
- Биндинги ролей хранятся в `project_role_bindings`; разрешение ролей определяется по subject/email/group/service.
- Run‑token идентичности обходят RBAC для минимального набора CP операций (артефакты/метрики DP).
- Все запреты фиксируются как `access.denied` в аудите.

## Обоснование
- Enforcement в каждом сервисе исключает скрытые правила и обеспечивает воспроизводимость решений.
- Project‑scoped RBAC соответствует матрице и упрощает аудит.
- Run‑token обход ограничивает DP доступ минимально необходимыми действиями.

## Последствия
- Требуется корректная конфигурация project‑resolver для всех новых API.
- Любые новые endpoints должны быть явно включены в RBAC‑проверки.
- Прямые роли в токене могут быть отключены конфигурацией (`AUTH_RBAC_ALLOW_DIRECT_ROLES`).
